# Test data & fixtures in cellpy-core

Durable record of how cellpy-core handles test data, and why. Established while
preparing the ground for the pandas→polars engine rewrite (issue #13).

## Context / problem

The compute engine (`summarizers.py`, `selectors.py`) needs to be tested on
**real** cycling data, not just synthetic frames. The realistic data lives in the
legacy `cellpy` repo — but it is stored as **instrument files** (`.res`, `.h5`,
Maccor/Neware/PEC exports) that require instrument loaders to read. The Arbin
`.res` path additionally needs an **ODBC / MS-Access driver** (Windows-only,
CI-hostile).

cellpy-core is the *core* engine and must stay **loader-free** and fast. So it
cannot read those files directly in its test suite.

## Decision

1. **Vendor parquet snapshots, not instrument files.** Real raw frames are
   exported once (where the loaders + ODBC exist) and committed as small
   `.parquet` under `tests/data/`. Parquet is engine-agnostic — it reads natively
   into both pandas and polars, which matters for the #13 migration.
2. **No instrument loaders as a cellpy-core dependency.** The test suite reads
   parquet directly; it never imports `cellpy`.
3. **Keep fixtures tiny.** A canonical multi-cycle fixture for golden parity plus
   a tiny (tens-of-rows) fixture for fast smoke tests. Current total ≈ 0.6 MB.
4. **Reproducible, not mystery binaries.** Every fixture is regenerated by
   `dev/regenerate_test_data.py` (two stages, documented in `tests/data/README.md`),
   so the provenance and exact transform are in version control.
5. **Provenance documented.** `tests/data/README.md` records the source files
   (cellpy / IFE, MIT) and the golden numbers.

## What we built (issue #13 prep)

- `dev/regenerate_test_data.py` — Stage A exports `data.raw` from cellpy files
  (run in cellpy's env); Stage B runs the cellpy-core engine on the exported raw
  and snapshots the step table (run in cellpy-core's env).
- `tests/data/arbin_cc_raw.parquet` — canonical 18-cycle Arbin raw (legacy
  `HeadersNormal` naming).
- `tests/data/arbin_cc_steps_expected.parquet` — frozen snapshot of the current
  engine's step table = **the regression oracle** for the rewrite.
- `tests/data/arbin_small_raw.parquet` — tiny 47-row/3-step Arbin-SQL-H5 raw.
- `tests/test_golden.py` — asserts the engine reproduces cellpy's published
  goldens (103 steps / 18 cycles / cycle-1 `data_point` 1457) **and** stays
  byte-faithful to the snapshot.

This gives #13 a real-data, cross-library oracle: the polars rewrite must keep
`test_golden.py` green.

## Why these source files

- `20160805_test001_45_cc_01.res` — the file cellpy's own suite asserts against,
  so its golden numbers are known-correct. Needs ODBC, hence pre-exported.
- `20200624_test001_cc_01.h5` (arbin_sql_h5) — smallest real cycling file, no
  ODBC; ideal fast fixture.
- `examples/data/maccor_two.txt` was **rejected** as a fixture: as parsed it has
  zero current and a single rest step, so it exercises nothing. A genuinely
  cycling non-Arbin fixture can be added later for format diversity.

## Alternatives considered

- *Read instrument files directly in cellpy-core tests* — rejected: pulls in
  loaders + ODBC, slow, non-portable, violates the loader-free core principle.
- *Synthetic data only* (`_helpers.create_raw_data`) — kept for unit tests, but
  insufficient as a correctness oracle; real data catches classification edge
  cases (e.g. the tiny fixture surfaces a step left with a blank `type`).
